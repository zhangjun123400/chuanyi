# 系统模特库管理设计说明

## 1. 目标

系统模特库从静态展示升级为可维护资产库，支持：

- 新增模特。
- 修改模特信息。
- 移除模特。
- 修改后同步到“新建试穿任务”的模特选择区。

## 2. 数据设计

当前 MVP 使用本地 JSON 存储：

- `model_assets`：用户新增或上传的模特资产。
- `model_library_changes`：默认系统模特的覆盖和隐藏记录。

这样做的原因是默认 Eva/Mia/Noah 仍然保留在代码种子里，但业务侧可以在界面上修改或隐藏它们。

生产数据库建议：

- `model_assets` 存储自定义模特。
- `model_library_changes` 存储系统默认模特的覆盖字段和删除状态。
- 删除采用软删除，避免历史任务找不到模特信息。

## 3. 后端接口

```text
GET    /v1/models/system
POST   /v1/models/system
PUT    /v1/models/system/{id}
DELETE /v1/models/system/{id}
```

说明：

- `GET` 返回系统默认模特、自定义模特和覆盖后的模特合集。
- `POST` 新增模特，需要上传真人/模特图片。
- `PUT` 修改模特，支持只改信息，也支持重新上传图片。
- `DELETE` 移除模特；如果有处理中的任务正在使用该模特，会拒绝移除。

## 4. 前端交互

系统模特库页面新增维护表单：

- 模特图片。
- 名称。
- 性别。
- 体型。
- 姿态。
- 适用品类。
- 风险标签。

模特卡片新增操作：

- 修改。
- 移除。

保存或移除后会重新加载模特列表，并同步刷新“新建试穿任务”里的“选择虚拟人/模特”板块。
