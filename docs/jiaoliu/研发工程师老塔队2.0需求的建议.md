# AI虚拟人穿衣SaaS PRD v2.0 用户系统 — 交流意见

> 基于 PRD 文档 `docs/prd/AI虚拟人穿衣SaaS_PRD_v2.0_用户系统.md` 的评审意见。

---

## 一、重点问题（建议修改）

### 1. 并发刷新Token的竞态条件

当 access token 过期，多个请求同时携带过期 token 触发刷新时，refresh token 会被重复消费或覆盖。PRD 描述的单一 refresh token 轮换策略在并发场景下存在风险：

- **问题**：请求A和请求B几乎同时到达，都检测到 access token 过期，都使用当前 refresh token 发起刷新。第一个成功后会作废旧 refresh token 并签发新的，第二个请求携带已作废的 refresh token 会失败，导致用户被强制登出。
- **建议**：后端对同一用户维度加锁（如 Redis 分布式锁或内存队列），或者前端采用 leader-follower 模式，由一个请求负责刷新、其他请求等待结果。

### 2. 两个 `auth.js` 文件命名易混淆

- **问题**：PRD 设计了 `backend/src/auth/auth.js`（鉴权中间件）和 `backend/src/admin/auth.js`（管理后台鉴权），同名不同目录，容易在 import 和调试时搞混。
- **建议**：考虑将 middleware 文件命名为 `auth-middleware.js` 或 `jwt-guard.js`，与管理员鉴权文件在语义上区分清楚。

### 3. 管理员默认密码打印到控制台（安全风险）

- **问题**：首次启动时将 `admin / ${随机字符串}` 打印到 stdout，若日志输出到云平台或持久化存储，密码会以明文保留在日志系统中。
- **建议**：改为写入独立文件（如 `.admin-credentials`），设置 600 权限，首次登录后提示修改密码；或仅在 tty 环境中输出并标记为一次性。

### 4. Token 黑名单存储增长

- **问题**：refresh token 撤销依赖内存黑名单（`revokedRefreshTokens`），每次轮换旧 token 入黑名单。服务重启后黑名单丢失，已撤销的 token 在重启后又能使用。
- **建议**：
  - 短期：将黑名单持久化到 SQLite 或文件，设置 TTL 定期清理过期条目。
  - 长期：使用 Redis，key 按 token 过期时间自动过期，天然解决存储增长问题。
  - 或考虑无状态方案：给 refresh token 加版本号，在用户表中存 `token_version`，轮换时不入黑名单而是递增版本号，验证时对照版本号。

### 5. 登录接口缺少频率限制

- **问题**：登录、刷新 token、忘记密码等接口未提及任何频率限制或防暴力破解措施。攻击者可无限制尝试用户名/密码组合。
- **建议**：对登录接口实施基于 IP + 账号维度的限流（如 5 次/分钟/IP，10 次/分钟/账号），连续失败后要求验证码或临时锁定。

### 6. 权限矩阵错误

- **问题**：PRD 权限矩阵中，operator 角色在「管理后台登录」列打了 ✓，但在文中明确说「管理员才能登录管理后台」。这存在矛盾。
- **建议**：修正权限矩阵，operator 的「管理后台登录」应标注为 ✗，与正文描述保持一致。

## 二、次要建议

### 7. 历史用户迁移的默认值

- PRD 提到「所有历史用户默认归类到某个默认租户」，建议明确命名约定，如 `tenant_default`，并在迁移脚本中增加幂等性处理。

### 8. 用户查询是否返回 `password_hash` 字段

- PRD 的用户查询接口未说明是否会过滤 `password_hash` 字段。强烈建议在序列化层统一排除敏感字段，避免误漏。

### 9. 邮箱是否允许修改

- PRD 使用邮箱作为唯一标识，但未明确说明用户是否可以修改邮箱。如需支持邮箱修改，需考虑唯一性校验、与 token 的解耦等问题。

### 10. SQLite 并发写入瓶颈

- PRD 选择 SQLite 存储用户数据，SQLite 在并发写入场景下较弱。如果预期并发用户数较高（如 50+ 同时操作），建议提前评估切换 PostgreSQL 的成本，或在架构上预留切换空间。

---

## 三、总结

整体 PRD 质量较高，体系完整，覆盖了双 Token JWT 认证、RBAC 权限模型、管理后台、多租户数据隔离等核心能力。以上 6 个重点问题建议在开发前修订，4 个次要建议可在迭代中逐步完善。
